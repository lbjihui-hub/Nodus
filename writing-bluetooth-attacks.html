Bluetooth Attack Surfaces: From Secure Simple Pairing to Real-World Exploitation

How Bluetooth connections are established at the protocol level, what attack surfaces vehicle infotainment systems expose, known CVEs with documented exploitation paths, and the tooling available to attackers including those with minimal technical sophistication.

The legacy problem — LMP PIN-based pairing

The original Bluetooth pairing method, known as LMP pairing or PIN-code based pairing, had a fundamental cryptographic weakness. The Link Key — the shared secret used to authenticate devices and derive encryption keys — was generated using three inputs: the Bluetooth Device Addresses of both devices, a 16-byte random number generated by the initiating device, and a PIN code entered by the user on both devices.

The weakness is in the PIN. Every input except the PIN was transmitted over the air in plain text during the pairing process. A passive sniffer capturing the exchange had all inputs except the PIN. Since a four-digit PIN produces only 10,000 possible values, an attacker could brute force every possible Link Key in seconds. Tools like the Ellisys Bluetooth Explorer could automate this entirely, determining the PIN and computing the Link Key from a passive capture with no user interaction required.

This was not a theoretical weakness. It was a practical attack that any attacker with a Bluetooth sniffer and basic tooling could execute.

Secure Simple Pairing and the Diffie-Hellman key exchange

Bluetooth 2.1 introduced Secure Simple Pairing, which replaced the PIN-based model with an asymmetric cryptographic approach based on Elliptic Curve Diffie-Hellman. The fundamental change was eliminating the PIN as the source of entropy for Link Key generation and replacing it with a key exchange that never transmits the critical secret over the air.

During an SSP pairing using the Numeric Comparison association model, both devices generate a key pair consisting of a public key and a private key. The public key is transmitted openly over the air. The private key never leaves the device that generated it under any circumstances.

Each device then performs a calculation using its own private key combined with the other device's public key. The mathematical property of elliptic curve cryptography guarantees that both calculations produce the same result even though neither device ever transmitted its private key. That result is the Diffie-Hellman shared secret, called the DHKey. It is a 192-bit number that both devices arrived at independently through their own separate calculations. An attacker capturing the entire exchange has both public keys but cannot compute the DHKey without one of the private keys. Deriving a private key from a public key on a 192-bit elliptic curve would require more computational time than the current age of the universe using every computer that exists today.

The six-digit number displayed on both devices during pairing is derived from the DHKey. Its purpose is man-in-the-middle protection. If an attacker intercepted the public key exchange and substituted their own keys, the DHKey calculations on both sides would produce different results and the displayed numbers would not match. The user confirming that both screens show the same number is literally performing the security verification step. After confirmation, the DHKey seeds the Link Key generation. The possible Link Key space is 2 to the power of 128, which is computationally infeasible to attack.

What a vehicle Bluetooth system actually accesses

The Honda HandsFreeLink system in a 2015 Civic uses Bluetooth BR/EDR and implements the Hands-Free Profile for calls, the Advanced Audio Distribution Profile for streaming, and the Phonebook Access Profile for contact synchronization. During initial pairing, the system explicitly asks whether to enable Phone Synchronization, which automatically imports the entire contact list and call history from the phone to the head unit's internal storage.

This means the Bluetooth connection supports OBEX — Object Exchange protocol — which is the file transfer layer that handles contact and calendar data. The contact list is stored persistently on the head unit and remains there after the phone disconnects. This data persists across sessions and is accessible to anyone who gains Bluetooth access to the head unit.

The reconnection attack surface

After initial pairing, reconnection happens automatically. When the car starts and the head unit powers on, it enters a page state and broadcasts inquiry packets seeking previously paired devices by their stored Bluetooth MAC addresses. When the phone's Bluetooth is enabled and comes within range, it responds and the connection is established using the stored Link Key for authentication — no user confirmation required.

This automatic trust relationship is the realistic attack surface. The head unit is advertising its MAC address and device name continuously while the car is running. An attacker within radio range has that information immediately. The practical range for BR/EDR is approximately 10 meters for Class 2 devices, though Class 1 amplified adapters extend this to 100 meters or beyond.

It is worth noting that for most Bluesnarfing and reconnection-based attacks, the Bluetooth radio in the head unit must be powered and active. The primary exposure window is when the vehicle is running and the system is broadcasting. However, BlueBorne class vulnerabilities operate differently — they exploit the Bluetooth stack implementation itself and require only that the radio be enabled, without any pairing or active connection.

Known CVEs and CVSS scores

CVSS scores run from 0 to 10. Scores from 0.1 to 3.9 are Low, 4.0 to 6.9 are Medium, 7.0 to 8.9 are High, and 9.0 to 10.0 are Critical. The score reflects attack vector, complexity, privileges required, user interaction required, and impact on confidentiality, integrity, and availability.

CVE-2019-9506 is the KNOB attack, Key Negotiation of Bluetooth, with a CVSS score of 7.0 High. This vulnerability targets the encryption key length negotiation phase that occurs after authentication. The Bluetooth specification allowed devices to negotiate the session encryption key length down to a minimum of one byte. An attacker within radio range could force both devices to accept a one-byte encryption key, reducing the brute force space to 256 possible values and enabling real-time decryption of all session traffic. This affects Bluetooth 2.1 through 5.1. Attack vector: adjacent network. Authentication required: none. User interaction required: none.

CVE-2020-26555 is a BR/EDR PIN pairing spoofing vulnerability with a CVSS score of 5.4 Medium. This affects legacy LMP pairing implementations and allows an attacker within radio range to impersonate a Bluetooth device during the PIN pairing process by exploiting the lack of public key authentication in legacy pairing. Attack vector: adjacent network. User interaction required: none.

CVE-2017-0785 is part of the BlueBorne attack suite with a CVSS score of 6.5 Medium. This specific CVE is an information disclosure vulnerability in the Android Bluetooth stack that does not require pairing or any user interaction. The Bluetooth radio simply needs to be enabled. BlueBorne as a broader suite also included remote code execution variants against Linux and Windows Bluetooth stacks, making it one of the most significant Bluetooth vulnerability disclosures to date. Attack vector: adjacent network. Authentication required: none. User interaction required: none.

CVE-2021-28139 is the BlueFrag vulnerability affecting Android 8 and 9, with a CVSS score of 8.8 High. Remote code execution via Bluetooth without pairing required. An attacker within radio range of a vulnerable Android device with Bluetooth enabled could execute arbitrary code silently. Attack vector: adjacent network. Authentication required: none. User interaction required: none.

Tools accessible to low-skill attackers

The barrier to entry for Bluetooth reconnaissance and exploitation is low. All of the following tools are available in Kali Linux by default or installable with a single apt command. An attacker with a laptop, a USB Bluetooth adapter capable of monitor mode, and a Kali Linux installation has access to the full toolkit.

hcitool and hciconfig are command line tools for Bluetooth device management and inquiry. Running hcitool scan returns the MAC addresses and device names of all discoverable Bluetooth devices within range. hcitool inq performs a raw inquiry returning the same information. This is device reconnaissance requiring no special privileges beyond Bluetooth adapter access.

btscanner is a passive Bluetooth device scanner that continuously monitors for discoverable devices and extracts device information including manufacturer, class, and supported services from the Extended Inquiry Response data. It requires no interaction with the target device.

obexftp is the tool used for Bluesnarfing attempts. It uses the OBEX file transfer protocol to connect to a target device's OBEX server and attempt to retrieve files including the phonebook at telecom/pb.vcf and call history at telecom/ch.vcf. Against vulnerable implementations this requires no authentication. The command syntax is straightforward and documented publicly.

Wireshark with a compatible Bluetooth adapter in monitor mode captures raw Bluetooth traffic for analysis. Combined with knowledge of the Link Key — obtainable if the attacker was present during initial pairing or if the KNOB attack successfully reduced the encryption key space — Wireshark can decrypt captured session traffic.

GATTacker and btlejuice target Bluetooth Low Energy implementations specifically, enabling man-in-the-middle interception of BLE connections. These are relevant for modern IoT devices and wearables rather than vehicle infotainment systems using BR/EDR.

Defensive posture

Disabling Bluetooth on your phone when not actively using it eliminates the automatic reconnection broadcast that creates the primary exposure window in public areas. Ensuring firmware updates are applied to both phone and vehicle head unit addresses known vulnerabilities including KNOB where patches exist. Being aware that your vehicle broadcasts its MAC address and device name continuously while running is the foundation of making an informed decision about acceptable risk in public environments.

The broader lesson applies across all wireless protocols. A system designed to connect automatically and synchronize data without friction will do so regardless of whether the initiating device belongs to the owner or to someone else within range. Understanding the protocol mechanics is what allows you to evaluate that risk accurately rather than treating wireless connectivity as either universally safe or universally dangerous.
